# ESP32-S3 联网交互终端项目书 

## 1. 项目目标
打造一个基于 ESP32-S3 的“小手机形态”联网终端，核心价值不是功能堆叠，而是展示可量产的嵌入式系统能力：
1. 稳定联网与协议实现（Wi-Fi/BLE + WebSocket/MQTT + HTTPS OTA）
2. 模块化驱动与应用架构（显示、触摸、音频、云通信解耦）
3. 可量化性能与功耗优化（CPU、内存、延迟、低功耗策略）
4. 工程化交付能力（Kconfig、组件化、日志、故障恢复）

## 2. 为什么这个项目适合 ESP32-S3  
该项目直接覆盖常见招聘要求：
1. `FreeRTOS` 多任务与双核调度
2. `ESP-IDF` 组件化开发与 `Kconfig` 配置管理
3. 外设驱动开发：`SPI LCD + Touch + I2S + GPIO`
4. 网络与安全：`TLS/HTTPS/WSS/MQTT`、断线重连、心跳机制
5. OTA 与可靠性：双分区升级、回滚、NVS 参数持久化
6. 性能调优：栈水位、堆占用、任务负载、功耗测试

## 3. 需求范围与边界
### 3.1 范围内
1. Wi-Fi 常驻联网，支持云端指令与状态上报
2. 本地 UI 与触摸交互（ILI9341 + XPT2046）
3. 云端 AI 接入（语音/文本通过服务端完成）
4. HTTPS OTA 在线升级

## 4. 系统架构（模块化）
建议按职责拆分为以下组件：
1. `src/wifi`：配网、重连、网络状态管理
2. `src/updata`：OTA 检查、下载、写分区、重启切换
3. `src/lcd`：屏幕和触摸驱动、基础 UI 输出
4. `src/cloud`：WSS/MQTT 客户端、心跳、重连
5. `src/proto`：业务协议编解码（msg_id/ack/retry）
6. `main`：仅做系统编排，不放具体业务逻辑

## 5. 协议设计原则（云端交互）
1. 统一消息头：`ver/dev_id/msg_id/ts/type`
2. 所有命令必须 `ack`，支持超时重传
3. 命令幂等，防止重复执行
4. 心跳与会话保活分离
5. 错误码标准化（参数错误、忙碌、网络失败等）

推荐优先级：
1. 第一阶段：`WSS + JSON`
2. 第二阶段：高吞吐路径迁移到二进制协议（可选）

## 6. RTOS 任务规划（示例）
1. `ui_task`：界面刷新与交互，固定帧率
2. `net_task`：网络收发与重连状态机
3. `proto_task`：协议解析、命令路由、ACK 管理
4. `ota_task`：升级流程独立执行
5. `sysmon_task`：CPU/heap/stack 监控与日志上报

双核建议：
1. Core0：UI、输入、轻量系统任务
2. Core1：网络、协议、音频流等高实时任务

## 7. 关键技术指标（可写进简历）
1. Wi-Fi 断线恢复时间：`< 3s`（同 AP 场景）
2. 云端命令到执行 ACK：`P95 < 150ms`（局域网）
3. 开机到联网可用：`< 5s`
4. OTA 成功率：`>= 99%`
5. 连续运行稳定性：`72h` 无崩溃
6. 低功耗待机电流：按硬件条件给出实测值与优化过程

## 8. 工程与质量要求
1. 所有模块独立头文件与 `CMakeLists.txt`
2. 配置项全部进入 `Kconfig.projbuild`
3. 错误处理统一返回 `esp_err_t`，禁止 silent fail
4. 日志分级：`I/W/E`，关键路径可关闭 debug
5. 每个里程碑都保留压测记录和问题复盘

## 9. 里程碑计划
1. `M1`：Wi-Fi + OTA + LCD/Touch 基线联通
2. `M2`：Cloud 协议（WSS/MQTT）+ ACK/重传机制
3. `M3`：语音链路接入（采集上传、云端返回、播放）
4. `M4`：低功耗与稳定性优化（长稳、重连、看门狗）
5. `M5`：简历版交付材料（架构图、指标图、关键代码说明）
